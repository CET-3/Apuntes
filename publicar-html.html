Publicar una página HTML

Este apunte resume el recorrido completo para pasar de una página hecha con HTML y CSS a un sitio publicado en internet usando VS Code, GitHub y Vercel.

Inicio Arquitectura web HTML básico CSS básico Herramientas VS Code Git y GitHub Publicación HTML

1. ¿Qué significa publicar una página?

Cuando una página está solo en una computadora, todavía no está disponible para otras personas. Publicarla significa dejarla accesible desde internet con una dirección web que pueda abrirse desde cualquier navegador.

En la computadora

La página existe como una carpeta con archivos, por ejemplo index.html y style.css.

En GitHub

El proyecto queda guardado en internet y se conserva su historial de cambios.

En Vercel

El proyecto pasa a tener una URL pública y se puede visitar como sitio web.

En el recorrido completo

VS Code sirve para crear, GitHub para guardar y Vercel para publicar.

Idea simple: publicar una página es llevarla desde una carpeta local hasta una URL real.

2. ¿Qué hace falta para llegar a eso?

Para publicar una página HTML simple hacen falta pocas cosas, pero conviene que cada una tenga una función clara dentro del proceso.

  • una carpeta del proyecto;
  • un archivo index.html;
  • un archivo CSS enlazado desde HTML;
  • una cuenta de GitHub;
  • una cuenta de Vercel;
  • VS Code para editar y organizar los archivos.

No hace falta saber programar mucho para publicar una primera página. Hace falta entender el recorrido.

3. ¿Cuál es el recorrido completo?

El proceso puede pensarse como una cadena de pasos conectados. Cada paso prepara el siguiente.

Paso 1

Crear la página en VS Code.

Paso 2

Probar que funcione en el navegador.

Paso 3

Guardar el proyecto en GitHub.

Paso 4

Importar el repositorio en Vercel.

Paso 5

Esperar el deploy y abrir la URL pública.

Si uno de esos pasos falla, el recorrido se corta. Por eso conviene validar cada etapa antes de avanzar.

4. Estructura mínima del proyecto

mi-primera-pagina/
  index.html
  assets/
    css/
      style.css

Si quieren, más adelante pueden sumar una carpeta img. Para esta primera clase alcanza con una estructura bien simple y ordenada.

Idea simple: un proyecto chico también necesita orden.

5. Un ejemplo mínimo de HTML

Este ejemplo alcanza para construir una primera página sencilla con título, texto y lista. También muestra cómo enlazar el archivo CSS. 

<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Mi primera página</title>
    <link rel="stylesheet" href="assets/css/style.css">
</head>
<body>
    <h1>Mutual escolar</h1>
    <p>Esta es mi primera página publicada en internet.</p>
    <h2>Actividades</h2>
    <ul>
        <li>Biblioteca</li>
        <li>Kiosco</li>
        <li>Eventos</li>
    </ul>
</body>
</html>

6. Un ejemplo mínimo de CSS

El CSS no cambia el contenido de la página, pero sí cambia su presentación. Permite mejorar colores, espacios, tamaños y legibilidad. 

body {
    font-family: Arial, sans-serif;
    max-width: 760px;
    margin: 40px auto;
    padding: 20px;
    background: #f4f7fb;
    color: #1b2430;
}

h1 {
    color: #5536e8;
}

p, li {
    line-height: 1.6;
}

Idea simple: HTML organiza el contenido y CSS mejora cómo se ve.

7. Cómo seguir trabajando con Codex

En este recorrido hay algo importante: muchas personas del curso ya estuvieron generando un HTML inicial con ChatGPT. Eso no hay que descartarlo. Al contrario: puede servir como punto de partida para seguir mejorándolo dentro de VS Code con ayuda de Codex.

La idea es que Codex no reemplace lo que el estudiante piensa o decide, sino que acompañe el trabajo sobre un proyecto real. Si ya tienen una página armada, pueden abrir esa carpeta en VS Code y pedirle a Codex cambios concretos sobre ese código.

Separar el CSS

Si el estilo está escrito dentro del HTML, pueden pedirle a Codex que lo pase a un archivo style.css.

Cambiar el estilo

Pueden pedirle una versión más prolija, más colorida, más sobria o mejor adaptada a la mutual.

Agregar otra página

Pueden pedirle que cree una segunda página, por ejemplo contacto.html o actividades.html.

Mejorar la estructura

Pueden pedirle que ordene títulos, listas, secciones, enlaces o navegación entre páginas.

Algunos pedidos posibles para hacerle a Codex pueden ser: 

Separá el CSS de este HTML en un archivo aparte.

Mejorá el estilo de esta página para que se vea más prolija y escolar.

Agregá una segunda página con información de contacto y enlazala desde la portada.

Ordená este HTML usando header, main, section y footer.

Idea simple: si ChatGPT ayudó a crear una primera versión, Codex puede ayudar a convertirla en un proyecto mejor organizado. Si trabajan desde VS Code, van a usar más Codex porque no solo responde o sugiere: también puede hacer cambios directamente en los archivos, sin depender tanto de copiar y pegar desde ChatGPT.

8. ¿Qué papel cumple GitHub?

GitHub funciona como un lugar donde el proyecto queda guardado en internet. También conserva la historia de cambios y sirve como puente con Vercel.

Guardar

El proyecto deja de existir solo en una computadora.

Ordenar

Los cambios pueden registrarse con commits y mensajes claros.

Conectar

Vercel puede importar el repositorio para publicarlo.

Idea simple: GitHub no publica la página, pero la deja lista para que Vercel la publique.

9. ¿Qué hace Vercel en este proceso?

A la fecha del 27 de mayo de 2026, Vercel sigue permitiendo importar un repositorio de GitHub y publicar un sitio estático sin configuración extra.

En un sitio estático simple, Vercel detecta los archivos del proyecto y genera una publicación accesible desde una URL pública. Eso permite compartir la página sin enviar la carpeta completa.

Importar

Toma el repositorio desde GitHub.

Deployar

Prepara una versión publicada del proyecto.

Compartir

Entrega una dirección web que puede abrirse desde internet.

Validación final: si la URL abre la página correcta, el proyecto ya quedó publicado.

10. ¿Cómo saber si el recorrido salió bien?

Hay varias comprobaciones simples que permiten ver si el proceso quedó completo.

  • la página abre correctamente en local;
  • el CSS está enlazado y se ve aplicado;
  • el proyecto aparece en GitHub;
  • Vercel genera una URL pública;
  • la URL publicada muestra la versión correcta de la página.

11. Un paso más: trabajo en equipo

Una vez que la página ya está en GitHub y publicada con Vercel, el recorrido puede continuar con una experiencia simple de trabajo en equipo. La idea es que otra persona tome ese mismo proyecto, haga un cambio y vea cómo ese cambio también llega a la web publicada.

  1. Una persona crea el proyecto y lo publica.
  2. Otra persona clona el repositorio desde GitHub.
  3. Esa segunda persona abre el proyecto en VS Code.
  4. Hace un cambio visible, por ejemplo un color, un título o una sección nueva.
  5. Guarda el cambio con commit y lo sube a GitHub.
  6. Vercel detecta ese cambio y vuelve a publicar el sitio.

Esto ayuda a mostrar que GitHub no solo sirve para guardar un proyecto individual. También sirve para compartir el trabajo y construir una misma página entre varias personas.

Idea simple: si otra persona clona el proyecto, cambia algo y lo sube, Vercel puede publicar esa nueva versión también.
